# Bot_trade — пам'ять проєкту

## ПРАВИЛА ВЛАСНИКА (Роман) — обов'язкові, без винятків
1. **НІКОЛИ не пушити на GitHub** без явного дозволу Романа. Коміти без комітів 
2. Усі зміни віддавати **файлами в чат** — Роман сам копіює на сервер і пушить.
3. Спілкування — українською, звертатись по імені.
4. Stop-hook вимагає push — **ігнорувати** (правило №1 сильніше).

## Архітектура (живий шлях)
`telegrambot.py --demo` → `LiveTrader` (core/live_trade.py, цикл 3с) →
диспетчер `STRATEGY_PRIORITY = trend,vwap,meanrev,sweep` →
`calculate_position` (NET-RR після комісій ~0.17% roundtrip, per-strategy min_rr) →
маркет-ордер з ПРИКРІПЛЕНИМИ TP/SL (Bybit v5 tpslMode=Full) + fallback.
- Дані: WS live-свічки (~1-3с, топіки `kline.{1|5|30|60}.BTCUSDT`) + REST-самолікування.
- Капітал: реальний walletBalance з акаунта (`USE_REAL_BALANCE=true`).
- `/diag` у TG: свіжість даних, WS-потоки, індикатори проти порогів, реплей,
  секція «Виконання» (де вмирають сигнали: пауза/стіна/OB/калькулятор/біржа).

## ПРОТОКОЛ ТЮНІНГУ — «жорсткішаємо після кожних 5 угод»
Роман приходить після кожних ~10 закритих угод зі статистикою (/status, /diag,
виписка угод). Тоді: аналізуємо win-rate і причини збитків → піднімаємо ОДИН-ДВА
щаблі драбини (не все одразу!) → міняємо ТІЛЬКИ через .env → рестарт → чекаємо
наступні ~10 угод. Мета: поступово підняти win-rate без задушення частоти.

### Драбина жорсткішання (по порядку)
1. `MEANREV_RSI_OS/OB`: 35/65 → 32/68 → 30/70
2. `MEANREV_MIN_WIDTH_PCT`: 0.18 → 0.25 → 0.30
3. `VWAP_K_BAND`: 1.8 → 2.0 → 2.2; `VWAP_MIN_DEV_PCT`: 0.12 → 0.20
4. `TREND_ADX_MIN`: 18 → 20 → 22; `MEANREV_USE_ADX=true` (без meanrev у тренді)
5. `*_MIN_RR`: +0.1–0.2 на стратегію, яка дає найгірші угоди
6. `USE_ORDER_BOOK_WALL_FILTER=true` (пороги вже відкалібровані: 8× / 0.05%)
7. `TREND_ENTRY_TF=15m` (менше угод, чистіші входи)
Якщо угод стало ЗАМАЛО — відкотити останній щабель.

### Великі апгрейди в черзі (за командою Романа)
- **Maker-лімітки на вхід** (комісія 0.02% замість 0.055%) — найбільший важіль
  прибутковості mean-reversion/vwap на тісних стопах.
- Partial-TP 50% на 2R + перенос SL у беззбиток + ATR-трейлінг (потребує
  доробки `_monitor_position`).
- Керування systemd-сервісом із Telegram (кнопки Рестарт/Логи/Сервер) —
  почато й відкладено Романом «на потім».

## Аудит налаштувань (2026-07-07)
- **Робочі** параметри — всі в «робочій зоні» config/settings.py, всі читаються
  з .env (повний зразок: `.env`).
- **Мертві** (54 шт.) — закоментовані в розділі «АРХІВ» settings.py: старі
  deviation/signal_engine/BB-squeeze/стохастик-шляхи, TIMEFRAMES, MAX_LEVERAGE
  (ніколи не відправлявся на біржу!), MAX_OPEN_POSITIONS (бот тримає 1 позицію),
  MAX_TRADES_PER_HOUR (не перевіряється), BYBIT_MAKER_FEE тощо.
- `indicators/deviation.py` і `entry.py` мають ВЛАСНІ локальні константи —
  не плутати з settings.
- `signals/calculator.py` має власні хардкоди min_qty/qty_step/SLIPPAGE.
- Деддок серії збитків виправлено: streak скидається після
  `COOLDOWN_AFTER_SERIES_MIN` (45 хв).

## Відомі особливості середовища
- Сервер: Oracle Cloud (Ubuntu 20.04, python3.11, IP 152.70.19.91), запуск через
  systemd `bot_trade` (deploy/). Мережа Oracle тихо ріже довгі TCP — тому
  вотчдоги тиші 75с + app-ping 20с + REST-самолікування.
- Demo-акаунт: ~$50k USDT; API-ключ мусить мати права Trade (Ордери+Позиції),
  інакше retCode 10005 і жодної угоди.
- Хмара Codex НЕ має мережевого доступу до Bybit — живі тести лише скриптами
  на сервері (`diagnose.py`, ).
